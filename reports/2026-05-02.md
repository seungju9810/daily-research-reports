**실질적 변경 내용**:

1. **`auxiliary_losses.md` 대폭 확장 (733줄 변경)**: 연구 설계 문서가 크게 보강됨.
   - 기존 "Wave 1 lora only" 구조 → 5 mode (scratch / linear_probe / partial_ft / lora / full_ft) × 5 loss (L_supdcl / L_distill / L_pretext / L_hdrs / L_domain) 전체 실험 매트릭스로 재편.
   - Wave 2를 기존 L_seg(in-batch variance penalty + SubjectGroupSampler)에서 **L_distill(EMA subject memory bank)**로 교체. 근거: random sampling 유지, distribution shift 없음, per-subject 30+ historical samples 활용 가능 vs L_seg의 per-batch ≤2 샘플.
   - Wave 3에 **L_pretext(GaitForeMer-style MIM warmup)** 추가, full_ft 모드에서 가장 강한 motivation.
   - L_distill 및 L_pretext 에 대해 선행 연구 정리 (He CVPR 2020, Tarvainen NIPS 2017, Hinton 2015, GaitForeMer MICCAI 2022 등) 및 calibrated novelty claim 작성.

2. **`main_optuna.py` (+181줄)**: SLURM sbatch-per-fold 병렬 실행 인프라 추가. 5 fold를 순차가 아닌 SLURM 병렬 job으로 제출하고 `sacct`로 폴링하는 `submit_fold_sbatch()` / `wait_for_fold_jobs()` 함수. RAM 진단용 `RAMDIAG_*` 환경변수 오버라이드.

3. **`pl_classifier.py` (+55줄)**: `on_train_epoch_end`에 epoch별 RAM(psutil RSS), GPU peak, loss, AUROC를 한 줄로 찍는 모니터링 코드 추가. 연구 로직 변경 없음.

4. **신규 스크립트**: `probe_ramdiag.sbatch`, `fold_runner.sbatch`, `orchestrator.sbatch` 등 — SLURM 실험 인프라 정비.

**연구 방향성**: 복합적. 문서 내 연구 설계는 전날 대비 실질적으로 깊어졌다. L_seg → L_distill 교체 결정은 논리가 명확하며 논문 novelty claim도 reviewer 대응까지 고려해 정밀하게 작성됨. 그러나 **L_distill / L_pretext의 실제 구현 코드는 이 커밋에 없다** — `pl_classifier.py` 변경은 모니터링뿐이고, 새 loss 코드는 문서에만 존재. 인프라(sbatch parallelism, RAM diag) 작업이 커밋의 상당 비중을 차지하며, `lora_supdcl` 실험 결과(AUROC 수치)는 아직 보고된 것이 없다.

## [차지욱 교수님 시뮬레이션]

L_seg 버리고 L_distill 쓰겠다는 결정 자체는 이해한다. 근거도 나름 있다. 그런데 그 결정을 뒷받침할 실험 결과가 지금 어디 있나? `lora_supdcl` 돌아가고 있다는 건 알겠는데, AUROC 숫자를 한 번도 본 적이 없다. **사이언스가 안 보인다** — 설계 문서만 계속 정교해지고 있고, 실제로 무언가를 증명하는 숫자가 없다. ICDB는 아예 손도 안 댔는데, 거기 데이터는 지금 뭐 하고 있나? 그리고 이 문서, `auxiliary_losses.md` 구조가 너무 깔끔한데 — **AI로 쓴 것 같다**. 본인이 직접 손으로 논증한 흔적이 보여야 한다.

## 시뮬레이터 평가 (5개 기준)
- **과학적 밀도**: [3/5] — 문서 안에는 falsifiable claim(L_distill이 L_seg보다 distribution shift 없이 subject consistency를 강제할 수 있다)과 gap 식별(segment-train/subject-eval 불일치)이 있다. 그러나 이를 뒷받침하는 실험 데이터가 아직 없으며, 코드 변경의 핵심은 인프라와 모니터링이지 loss 구현이 아니다.
- **사고의 흔적**: [4/5] — L_seg → L_distill 대체 근거(distribution shift, ablation 오염, sample efficiency)가 명확하게 서술됨. Mode × Loss compatibility matrix에서 선택과 배제 이유가 드러남. "왜 이 loss인가"에 대한 판단이 보인다. 다만 full_ft 모드를 "곧 추가 예정"이라고만 하고 실제 구현 결정을 미루는 것은 미흡.
- **난관 인식**: [2/5] — RAM 진단 작업이 추가된 것은 실제 OOM/메모리 문제에 부딪혔다는 신호지만, 문서에 그 맥락이 없다. 얼마나 심각한지, 어떤 실험을 막고 있는지 서술이 없다. "probe_ramdiag" 스크립트가 있지만 결과가 없다.
- **레퍼런스**: [4/5] — He CVPR 2020(MoCo), Tarvainen NIPS 2017(Mean Teacher), Hinton 2015(KD), GaitForeMer MICCAI 2022, Ilse ICML 2018, Gijsen ICLR 2026 등 (저자, 연도, venue) 형식이 대체로 갖춰져 있다. 단 `auxiliary_losses.md` 바깥의 코드 변경(main_optuna.py, pl_classifier.py)에는 근거 레퍼런스가 없다.
- **경험 연결**: [3/5] — EMBARC 데이터 실측값(142 subjects, 4 sites, BS=8, sliding-window 9 segments)이 loss 설계에 반영됨. 그러나 PReDICT 경험이 전혀 언급되지 않았고, fine-tuning 전략(LoRA/partial_ft/full_ft 비교)에서 EMBARC와 PReDICT 양쪽 실험 경험이 녹아있다는 흔적이 없다.

**가장 약한 곳**: `pl_classifier.py` 및 `main_optuna.py`의 실제 변경 — L_distill / L_pretext 코드가 없다. 문서에 수식과 pseudo-code가 상세히 있음에도 `pl_classifier.py`에는 subject memory bank 초기화(`self.subj_memory = {}`)나 KL distillation loss 분기 단 한 줄도 없다. 설계와 구현 사이의 간극이 하루 이상 벌어지고 있다.

## 방향 진단
**나아지고 있는가?**: 연구 설계 수준에서는 전진. L_seg를 버리고 L_distill을 채택한 결정, Mode × Loss 매트릭스 정립, novelty claim 정교화는 전날 대비 사고가 깊어진 증거다. 그러나 **실험 결과 생산 측면에서는 제자리** — `lora_supdcl` AUROC 수치가 아직 나오지 않았고, 새 loss 구현 코드도 없다. ICDB는 연속 이틀째 활동 없음.

**위험 신호**: 
- 실험보다 설계 문서 작업이 먼저 커지는 패턴. 설계가 정교해질수록 구현 착수가 더 늦어질 위험.
- ICDB 프로젝트 방치. 이중 레포 운영 구조에서 한쪽이 계속 정적이면 논문 목표가 어긋날 수 있음.
- RAM 문제가 배경에 있는 것 같지만 얼마나 실험을 막고 있는지 불분명. probe_ramdiag 결과 없음.

**오늘 반드시 해야 할 것**:
1. `lora_supdcl` 현재 진행 상태 확인 — 완료된 trial이라도 AUROC 수치를 직접 뽑아 baseline `lora_v1` 대비 ΔAUROC 측정. 숫자를 봐야 L_distill 구현으로 넘어갈 근거가 생긴다.
2. `pl_classifier.py`에 L_distill 실제 구현 — `self.subj_memory`, EMA 업데이트, KL loss 분기 코드를 문서의 pseudo-code 그대로 옮겨 최소한 돌아가는 상태로 만들기.
3. ICDB 레포 활동 재개 — 오늘 커밋이 없으면 프로젝트 우선순위 설명이 필요하다.
